The Ingredient class

It is the abstract base class (ABC) for burger ingredients. We only have 3 classes in this class hierarchy and Ingredient sits at the top. To give you an overview of the class design, here is the class diagram:

In the diagram, + denotes the public members while - denotes the private members.

Details will be provided below. For anything that appears to be unclear to you, you should refer to the given code if they are available. First, let's look at this ABC: the Ingredient class.

Data members

int freshness = 100;

This is the freshness of the ingredient. It is always set to 100 initially and will decrease by 1 per update, which will be described later.

Member functions

virtual ~Ingredient();

This is the destructor. It doesn't really do anything in this class, but as a good practice we should always declare it virtual in an ABC. If you are curious why, think about what may happen if the base class destructor is not virtual when we delete a derived class object via a base class pointer.

int getFreshness();

It returns the freshness.

bool isSpoiled();

It returns whether the ingredient is spoiled, that is, has a freshness of 0.

virtual void update();

Every hour in our simulation, the update function of all ingredients will be called to update their states. For all ingredients, their freshness values have to each be decreased by 1 (lower-bound by 0). For specific ingredients, the update function may be overridden to specify additional state updates required.

virtual void print() = 0;

It prints the information of the ingredient. It is made pure virtual in this ABC and shall be overridden in all concrete derived classes.

The Bun class

It is a bun. It is a concrete derived class of Ingredient. Each Happy Teemo Burger requires two of it.

Data members

int softness;

This is the softness of the bun. It is set by the constructor and will also decrease every update.

Member functions

Bun(int softness)

It is the constructor. It sets the initial softness.

int getSoftness();

It returns the softness.

void update() override;

Since a bun is an ingredient, its freshness must also be updated accordingly in an update. Additionally, its softness is decreased by 2 (lower-bound by 0).

You need to implement this function.

void print() override;

It prints the information of the bun. You should read the given implementation and the sample output to see what it prints.

The Meat class

It is a piece of meat. It is a concrete derived class of Ingredient. Each Happy Teemo Burger requires one of it.

Data members

int doneness;

This is the doneness of the meat. (i.e. a gauge of how thoroughly cooked the meat is) It is set by the constructor and will simply remain the same forever.

Member functions

Meat(int doneness)

It is the constructor. It sets the doneness.

int getDoneness();

It returns the doneness.

void print() override;

It prints the information of the meat. You should read the given implementation and the sample output to see what it prints.

The BurgerShop class

It is a burger shop. Specifically, it models how the Happy Teemo Burgers shop is run.

Data members

Ingredient** ingredientStorage;

It models the ingredient storage of the burger shop. The storage can store buns and meats.

Implementation-wise, this is a dynamic array of ingredient pointers which each point to either a Bun dynamic object, or a Meat dynamic object, or nothing (i.e. nullptr).

int ingredientStorageCapacity;

This is the ingredient storage capacity. That is, the maximum numbers of ingredients the storage can hold. That essentially is the size of the ingredientStorage array. We can also say that the ingredient storage has ingredientStorageCapacity storage slots.

int ingredientStorageUsed = 0;

This is the number of storage slots currently used in the ingredient storage for storing buns and meats. Naturally, it will never exceed the ingredientStorageCapacity. It is always 0 initially.

int burgerServed = 0;

This counts how many burgers have been successfully crafted and served to Teemo's beloved customers. It is always 0 initially.

Member functions

BurgerShop(int ingredientStorageCapacity);

It is the constructor. It initializes the ingredientStorageCapacity and ingredientStorage.

~BurgerShop();

It is the destructor. All remaining dynamic objects created in this class should be deallocated properly.

You need to implement this function.

void attemptToMakeBun(int softness);

It attempts to make a bun and store that for future use. As you can see in the given code, it calls the makeBun function which will be described later.

void attemptToMakeMeat(int doneness);

It attempts to cook a piece of meat and store that for future use. As you can see in the given code, it calls the makeMeat function which will be described later.

void attemptToMakeAndServeBurger(int requiredBunSoftness, int requiredMeatDoneness);

It attempts to craft a Happy Teemo Burger using the stored ingredients. As you can see in the given code, it calls the makeAndServeBurger function which will be described later.

void update();

It updates the states of all ingredients by invoking their own update functions. It will be called every hour in the simulation.

void print();

It prints the information of the shop, in particular its storage. You should read the given implementation and the sample output to see what it prints.

bool isStorageFull();

It returns whether the ingredient storage is full. The storage is full when there is no empty storage slot left.

void addFoodToStorage(Ingredient* food);

It adds the given ingredient to the first non-empty slot in the ingredient storage. It may be used in the makeBun and makeMeat functions.

bool makeBun(int softness);

It first checks if the storage is full. If yes, it does nothing and returns false.

Otherwise, it proceeds to create a new Bun object of the specified softness, adds that to the first non-empty ingredient storage slot, and returns true.

bool makeMeat(int doneness);

It first checks if the storage is full. If yes, it does nothing and returns false.

Otherwise, it proceeds to create a new Meat object of the specified doneness, adds that to the first non-empty ingredient storage slot, and returns true.

bool makeAndServeBurger(int requiredBunSoftness, int requiredMeatDoneness);

Well, finally, it is time to make burgers!

The challenge is, our beloved customers have specific requests on exactly how their burgers should be made, as specified by requiredBunSoftness and requiredMeatDoneness.

Teemo follow these rules strictly (and so will you ;)):

  • A Happy Teemo Burger uses exactly 1 suitable meat and 2 suitable buns. If we cannot gather from the storage all those 3 specific ingredients for a burger, the function will simply return false.
  • Regarding what the suitable ingredients are:
    • Spoiled ingredients should never be used. By the way, they just stay in the storage forever because Teemo is apparently too busy. Don't help him to remove those spoiled stuff yourself.
    • A suitable meat must have at least requiredMeatDoneness of doneness.
    • A suitable bun must have at least requiredBunSoftness of softness.
  • When looking for each of those 3 ingredients, Teemo simply always picks the first suitable meat and first two suitable buns from the storage - i.e. technically, the smaller the array index the better for each of them.
  • After we have found the 3 suitable ingredients, we do the following to actually craft and serve the burger:
    • Remove the used ingredients from the storage. That means you should dynamically deallocate the 1 dynamic Meat object and the 2 dynamic Bun objects, and then for each of them set the corresponding ingredientStorage[i] to nullptr indicating that the originally-occupied storage slot is now empty and thus reusable later for new ingredients.
    • Since 3 ingredients have been used, decrease ingredientStorageUsed by 3.
    • Add 1 to burgerServed as our now-satisfied customer walks away with our burger happily.
    • Return true.

You need to implement this function.

Sample output & grading

Your finished program should produce the same output as our sample output. Please note that sample output, naturally, does not show all possible cases. For this assignment, we will also check if your program has any memory leak. We will be using the following command to compile and run your program under the "Test environment" we mentioned in tutorial 1, assuming your source files are located at "csci3180/a1".

g++ -std=c++20 -fsanitize=address,leak -o main csci3180/a1/*.cpp && ./main

It will report detected memory leak as runtime errors when you run the program (i.e. extra output indicating that there is a memory leak). If you want to learn about it, you may read this manual. With this said, we will still try to grade your program output (without leak checking, that is, with command g++ -std=c++20 -o main csci3180/a1/*.cpp && ./main) after applying some mark penalty even if there is a memory leak detected, so do your best in other tasks anyway if memory leak is too hard to fix for you. Please note that, on the other hand, if your program cannot even be compiled or simply crashes on our grading environment, no mark can be given. Be aware that C++ programs may behave differently in different environments. For grading, we will only consider your program output on our grading machine using the official grading environment. Therefore, while you are free to work out your solution in any environment, do test it in our official grading environment and by all means avoid compilation errors and crashes before your submit your work. Program stability is king.
